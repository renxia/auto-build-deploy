提交 PR 指南

学习如何正确提交 Pull Request

PR 流程

从准备到合并的完整流程

步骤 1

准备工作

确保代码质量和完整性

  • 运行所有测试并确保通过
  • 检查代码风格和 ESLint 规则
  • 更新相关文档
  • 确保提交信息清晰
步骤 2

创建 PR

在 GitHub 上创建 Pull Request

  • 推送分支到个人仓库
  • 创建 Pull Request
  • 填写 PR 模板
  • 添加相关标签
步骤 3

代码审查

参与代码审查过程

  • 响应审查意见
  • 修改代码问题
  • 更新测试用例
  • 解决冲突
步骤 4

合并发布

PR 被接受并合并

  • 通过所有检查
  • 获得审查批准
  • 合并到主分支
  • 清理分支

PR 检查清单

提交前的必要检查项目

代码质量

  • 所有测试通过
  • 代码覆盖率达标
  • 遵循编码规范
  • 没有 ESLint 错误
  • 性能影响评估
  • 安全性检查

文档更新

  • API 文档更新
  • README 文件更新
  • 变更日志记录
  • 示例代码更新
  • 配置说明更新
  • 迁移指南（如需要）

测试覆盖

  • 新功能有测试
  • 边界情况测试
  • 错误处理测试
  • 集成测试更新
  • 性能测试（如需要）
  • 回归测试通过

兼容性

  • 向后兼容性
  • 跨平台兼容
  • 版本依赖检查
  • 破坏性变更标记
  • API 变更说明
  • 升级路径清晰

提交信息规范

标准化的 Git 提交信息格式

# Git 提交信息规范

## 格式
<type>(<scope>): <subject>

<body>

<footer>

## 类型 (type)
- feat: 新功能
- fix: 修复 Bug
- docs: 文档更新
- style: 代码格式（不影响功能）
- refactor: 代码重构
- test: 测试相关
- chore: 构建过程或辅助工具的变动
- perf: 性能优化
- ci: CI/CD 相关

## 范围 (scope)
- cli: 命令行界面
- core: 核心功能
- api: API 相关
- config: 配置相关
- docs: 文档
- test: 测试

## 示例
feat(cli): add new command for user management

Add support for creating, updating, and deleting users
through the CLI interface.

- Add create-user command
- Add update-user command  
- Add delete-user command
- Update help documentation

Closes #123
Fixes #456

## 规则
1. 使用英文编写
2. 首行不超过 50 字符
3. 正文每行不超过 72 字符
4. 使用现在时态
5. 首字母小写
6. 不要以句号结尾

PR 模板

标准的 Pull Request 模板

## 变更描述
简要描述此 PR 的变更内容和目的。

## 变更类型
请删除不适用的选项：
- [ ] Bug 修复（不破坏现有功能的变更）
- [ ] 新功能（添加功能的变更）
- [ ] 破坏性变更（会导致现有功能不工作的修复或功能）
- [ ] 文档更新
- [ ] 性能改进
- [ ] 代码重构
- [ ] 依赖更新

## 测试
描述你如何测试了这些变更：
- [ ] 单元测试
- [ ] 集成测试
- [ ] 手动测试
- [ ] 现有测试通过
- [ ] 添加了新的测试

## 检查清单
- [ ] 我的代码遵循项目的代码风格
- [ ] 我已经进行了自我审查
- [ ] 我已经添加了必要的注释
- [ ] 我已经添加了测试来证明我的修复有效或功能正常工作
- [ ] 新的和现有的单元测试都通过了
- [ ] 我已经更新了相关文档
- [ ] 我的变更不会产生新的警告
- [ ] 我已经检查了我的代码，没有拼写错误

## 相关 Issue
修复 #(issue 编号)
关联 #(issue 编号)

## 破坏性变更
如果此 PR 包含破坏性变更，请描述：
- 什么被改变了
- 为什么需要这个变更
- 如何迁移现有代码

## 截图（如果适用）
添加截图来帮助解释你的变更。

## 附加信息
添加任何其他相关信息。

常见问题解决

PR 过程中的常见问题和解决方案

合并冲突

分支与主分支存在冲突

解决步骤：

  1. 1拉取最新的主分支代码
  2. 2解决冲突文件
  3. 3重新测试修改的代码
  4. 4推送解决冲突后的代码

命令示例：

git fetch upstream
git rebase upstream/main
# 解决冲突
git add .
git rebase --continue
git push --force-with-lease

CI 检查失败

自动化检查未通过

解决步骤：

  1. 1查看 CI 日志找出问题
  2. 2本地复现并修复问题
  3. 3更新测试用例
  4. 4重新推送代码

命令示例：

npm run lint
npm run test
npm run build
git add .
git commit -m "fix: resolve CI issues"
git push

代码审查反馈

审查者提出修改建议

解决步骤：

  1. 1仔细阅读审查意见
  2. 2逐一回应每个建议
  3. 3修改代码并解释原因
  4. 4请求重新审查

命令示例：

# 修改代码
git add .
git commit -m "address review feedback"
git push
# 在 GitHub 上回复审查意见

测试覆盖率不足

新代码缺少测试

解决步骤：

  1. 1分析覆盖率报告
  2. 2为未覆盖代码添加测试
  3. 3确保边界情况测试
  4. 4验证测试质量

命令示例：

npm run test:coverage
# 查看覆盖率报告
# 添加测试文件
npm run test
git add .
git commit -m "test: improve test coverage"

最佳实践

提高 PR 质量的建议

保持 PR 小而专注

每个 PR 只解决一个问题

优势：

  • 更容易审查
  • 降低出错风险
  • 加快合并速度
  • 便于回滚

编写清晰的描述

详细说明变更内容和原因

优势：

  • 帮助审查者理解
  • 记录变更历史
  • 便于后续维护
  • 提高沟通效率

及时响应反馈

积极参与代码审查讨论

优势：

  • 加快审查进度
  • 提高代码质量
  • 增进团队协作
  • 学习最佳实践

保持代码同步

定期同步主分支代码

优势：

  • 减少合并冲突
  • 及时发现问题
  • 保持代码最新
  • 简化合并过程
